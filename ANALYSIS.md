## 🔴 P0 — Blocks Compilation / Runtime Crash

### 1. Missing Firebase Dependencies (37 errors)

Three packages are imported but **not listed** in `pubspec.yaml`:

| Package | Files affected |
|---------|---------------|
| `firebase_core` | `lib/core/services/notification_service.dart` |
| `firebase_messaging` | `notification_service.dart`, `lib/providers/notifications_provider.dart` |
| `flutter_local_notifications` | `notification_service.dart` |
| `firebase_options.dart` | `notification_service.dart` (generated file, doesn't exist) |

**Root cause:** `pubspec.yaml` is missing these dependencies. `firebase_options.dart` is auto-generated by `flutterfire configure` — that command was never run.

**Fix:** Either add the packages (`flutter pub add firebase_core firebase_messaging flutter_local_notifications`) and run `flutterfire configure`, or wrap the Firebase-dependent code behind a conditional import/abstract interface so the app compiles without Firebase.

## 🟠 P1 — Security & Production Critical

### 1. Tokens Stored in SharedPreferences (Plaintext)

JWT access and refresh tokens are persisted via `PrefsService` using `SharedPreferences` — unencrypted on disk.

**Fix:** Replace with `flutter_secure_storage`:
```dart
// Add dependency: flutter pub add flutter_secure_storage
// Replace PrefsService.saveTokens/getAccessToken/getRefreshToken/clearTokens
```

### 2. No Dark Mode

Only `AppTheme.light` exists in `lib/core/theme/app_theme.dart`. No dark `ThemeData` is built.

**Fix:** Add `AppTheme.dark` getter and wire into `MaterialApp` via `themeMode` listening to a system/theme preference provider.

### 3. LogInterceptor Prints to Console in Production

`ApiClient` enables `LogInterceptor` with `requestBody: true, responseBody: true` — this leaks sensitive data (tokens, passwords, PII) to console logs.

**Fix:** Make logging conditional on build mode:
```dart
if (!kReleaseMode) { _dio.interceptors.add(LogInterceptor(...)); }
```

### 4. No Certificate Pinning

All HTTPS traffic uses `Dio` with default trust — vulnerable to MITM if a CA is compromised.

**Fix:** Add certificate pinning via Dio's `BadCertificateHandler` or platform-native pinning.

### 5. No Biometric Auth for Clock-In

Clock-in only checks GPS geofence — no biometric verification. A stolen unlocked device can be used to clock in.

**Fix:** Add optional biometric check via `local_auth` package before clock-in.

### 6. No `Accept-Language` Header

`ApiClient` never sends the user's locale preference to the backend, so error messages and labels can't be locale-aware.

**Fix:** Read locale from `PrefsService.getLanguageCode()` and set `Accept-Language` header:
```dart
final locale = await PrefsService.getLanguageCode() ?? 'en';
options.headers['Accept-Language'] = locale;
```

### 7. Navigation Routes Hardcoded in Notification Service

Notification deep-linking routes (`/shifts`, `/absences`, `/announcements`, `/tasks`, `/notifications`) are used in `NotificationService._routeFromData()` but are never defined in `main.dart`'s route table.

**Fix:** Either define these named routes in `MaterialApp.routes` or use `Navigator.pushNamed` with route constants.

### 8. `PrefsService.logout()` Removes Tokens Twice

Lines 91-93 in `prefs_service.dart` duplicate the `remove` calls:
```dart
await prefs.remove(_kAccessToken);  // line 91
await prefs.remove(_kRefreshToken); // line 92
await prefs.remove(_kAccessToken);  // line 93 — duplicate!
await prefs.remove(_kRefreshToken); // line 94 — duplicate!
```

**Fix:** Remove the duplicate lines.

## 🟡 P2 — Feature Incompleteness

### 1. Only 1 Smoke Test

`test/widget_test.dart` contains a single 17-line smoke test that just verifies the app renders without crashing. Zero unit tests, widget tests, or integration tests.

**Critical test gaps:**
- `AuthProvider.signIn/signOut/tryRestoreSession` — no tests
- `ClockProvider.clockIn/clockOut/startBreak/endBreak` — no tests
- `AbsenceProvider.loadAbsences/submitAbsence` — no tests
- All repositories — no tests for API call formatting or error mapping
- All models — no `fromJson`/`toJson` roundtrip tests
- All pages — no widget tests

### 2. No Offline Support

Zero local caching. Every data fetch goes directly to the API. If the network is down:
- Login fails entirely
- Dashboard shows nothing
- Clock-in/out fails
- Schedule, absences, announcements all show loading errors

**Fix:** Add local caching layer (Hive/Isar/Drift) with offline-first pattern. At minimum, cache the last known good state and show it with a "stale data" indicator when offline.

### 3. Duplicate Location Models

Two classes model the same concept with incompatible types:

| `WorkLocationModel` (16 fields) | `WorkplaceLocation` (6 fields) |
|---------|---------|
| Used by `ClockProvider` | Expected by `LocationCard` |
| Has `locationCode`, `locationType`, `timezone`, etc. | Minimal: name, address, lat/lng, radius, buffer |
| `fromJson` reads snake_case keys | `fromJson` reads camelCase keys |

**Fix:** Merge into a single model. `WorkLocationModel` is more complete — update `LocationCard` to accept it.

### 4. Backend JSON Alignment Gaps

**Enum serialization:** Flutter `fromJson` expects camelCase (`vacation`, `sickLeave`) but backend returns UPPER_SNAKE (`VACATION`, `SICK_LEAVE`). Same for statuses.

**JSON key casing:**
| Flutter reads | Backend likely returns |
|--------------|----------------------|
| `first_name`, `last_name` (user) | `firstName`, `lastName` |
| `startTime`, `endTime` (shift) | `start_time`, `end_time` |
| `access_token` (auth) | `accessToken` or `access_token` |

**Missing computed fields:** `working_days` is expected by `AbsenceModel` but absence API likely doesn't compute it.

### 5. Hardcoded Dashboard Announcements

`_AnnouncementList._items` in `dashboard.dart:323-345` contains 3 hardcoded English announcements. These are:
- Not localized (no German versions)
- Not fetched from the API
- Not reactive to the `AnnouncementProvider`

**Fix:** Wire the `AnnouncementProvider` into the dashboard and remove the static list.

### 6. Absence Submit Uses Wrong Endpoint

`ApiAbsenceRepository.submitAbsence()` posts to `ApiConfig.absenceCalendar` which is intended for calendar viewing, not creation.

**Fix:** Add a dedicated create endpoint in `ApiConfig`:
```dart
static const String absenceCreate = '/api/absences/calendar'; // or /api/absences
```

### 7. No Pull-to-Refresh

No page implements `RefreshIndicator`. Users can't refresh data without navigating away and back.

### 8. No Crash Reporting / Monitoring

No Sentry, Firebase Crashlytics, or any error monitoring integrated. Crashes and errors are silent unless a user reports them.

### 9. No Analytics

Zero telemetry or analytics. No way to measure DAU, feature usage, drop-off, or conversion.

### 10. No CI/CD Pipeline

No GitHub Actions, GitLab CI, or any automated pipeline for:
- Running `flutter analyze` on PRs
- Running tests on every commit
- Building APK/IPA artifacts
- Deploying to app stores

### 11. `withOpacity()` Deprecated (~50 occurrences)

`Colors.black.withOpacity(0.06)` is deprecated in favor of `Colors.black.withValues(alpha: 0.06)`. ~50 uses across the codebase.

### 12. Legacy Theme File

`lib/core/theme.dart` (24 lines) duplicates constants that already exist in `lib/core/theme/` directory. Should be deleted to avoid confusion.

### 13. Clock Provider Timer Bug

`ClockProvider` uses a single field `_sessionTimer` for both session and break timers:
- `_startSessionTimer()` → assigns `_sessionTimer` for work time
- `_startBreakTimer()` → also assigns `_sessionTimer` for break time
- `_stopSessionTimer()` → cancels `_sessionTimer`

On break transitions, the previous timer reference is lost before being cancelled, potentially leaking timers or corrupting accumulated time.

**Fix:** Use separate `_sessionTimer` and `_breakTimer` fields.

### 14. No Pagination for Shifts and Absences

While the API supports cursor-based pagination (announcements, notifications), the shift and absence endpoints/lack cursor handling in providers. Large datasets will cause unbounded memory growth.

### 7. Auth Interceptor Anti-Pattern

`_AuthInterceptor.onRequest()` mutates `options.headers['Authorization']` after calling `handler.next(options)`. The interceptor should set the header before calling `handler.next()`, not after.

`NoOpNotificationRepo` is used while logged out — correct for avoiding 401 errors, but means FCM tokens are not registered until after the first notification provider re-creation on login.

`AppProviders.build()` uses `FutureBuilder` to initialize `LocaleProvider` and `NotificationService` simultaneously. During the future, a raw `MaterialApp` with a loading spinner is rendered as a fallback — this can cause a flash of empty content.
